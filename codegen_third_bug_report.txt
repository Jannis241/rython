Rython Codegen Bug Report - dritter Durchlauf
Datum: 2026-05-14
Scope: rython_to_ir Codegen und direkt angrenzende Parser/IR-Stellen, soweit sie bereits
als Feature verdrahtet sind. Reine TODOs wie fehlendes ir_to_asm sind nicht enthalten.

Aktueller Check:
- cargo check -p rython_to_ir: erfolgreich
- Warnings bleiben: zwei unused imports, unused Variable.name, AST-Feld Op nicht snake_case

Kurzfazit:
Die aktuellen groessten echten Bugs sind:
1. Variants werden semantisch falsch als int statt als Variant-IR-Wert erzeugt.
2. Inline-ASM kann Werte lesen, aber nicht sauber in Rython-Variablen zurueckschreiben.
3. Operator-Overloads umgehen die neue Funktionssignatur-Pruefung.
4. Unaere Operator-Overloads werden falsch klassifiziert.
5. Name-Mangling/Preprocessing erkennt Kollisionen und Duplikate nicht.
6. Field-Access funktioniert nur auf lvalues, nicht auf temporaren Struct-Werten.


1) Variant-Literals werden als int lowered statt als Variant
Stelle:
- crates/rython_to_ir/src/codegen/expr.rs:54-67
- crates/rython_to_ir/src/ir.rs:133-138

Was passiert:
gen_variant_literal sucht den Case-Index und ruft dann self.gen_intliteral(...)
auf. Dadurch wird z.B. Color::Red zu einem I64-Wert mit 0.

Aktueller Kern:
  let case_idx = cases.iter().position(|c| c == case_name)?;
  self.gen_intliteral(&case_idx.to_string())

Warum das ein Bug ist:
Im IR existiert bereits IrInstruction::InitVariant:
  InitVariant { temp_id, ty: IrType::Named("Color"), case_name: "Red" }

Aber diese Instruction wird nicht benutzt. Dadurch hat Color::Red den Typ I64,
nicht Named("Color"). Folgender Code sollte funktionieren, scheitert aber
typisch an MismatchedTypes oder verliert die Typinformation:

  variant Color { Red, Blue }

  fn main() {
      let c: Color = Color::Red;
  }

Erwartetes Verhalten:
gen_variant_literal sollte einen Temp erzeugen, InitVariant emittieren und
(temp, IrType::Named(variant_name)) zurueckgeben.

Fix-Richtung:
  let temp = self.next_temp_id();
  self.block_handler.add_instruction_to_current_block(
      IrInstruction::InitVariant {
          temp_id: temp,
          ty: IrType::Named(variant_name.clone()),
          case_name: case_name.clone(),
      }
  )?;
  Ok((temp, IrType::Named(variant_name.clone())))

Zusatz:
Falls Variants spaeter Payloads bekommen sollen, reicht InitVariant mit case_name
allein nicht mehr. Fuer die aktuellen cases ohne Payload waere das aber korrekt.


2) Inline-ASM behandelt alle %variablen nur als Input, nie als Output
Stelle:
- crates/rython_to_ir/src/codegen/stmt.rs:81-139

Was passiert:
substitute_asm_variables ersetzt jedes %name durch ein neues Temp, indem es die
Variable laedt:

  let var = self.lookup_variable(name)?;
  let temp = self.next_temp_id();
  IrInstruction::Load { temp_id: temp, ty: var_ty, addr: var_addr }
  out.push('%');
  out.push_str(&temp.0.to_string());

Beispiel:
  let start_pointer: int = 0;
  asm { mov %start_pointer, rax }

Der Generator erzeugt sinngemaess:
  %x = load start_pointer
  asm "mov %x, rax"

Warum das ein Bug ist:
Wenn %start_pointer in ASM als Ziel/Registerersatz benutzt wird, wird nur der
alte Wert geladen. Es gibt danach keinen Store zurueck in start_pointer.
Das heisst: ASM kann aktuell nicht zuverlaessig Rython-Variablen setzen.

Erwartetes Verhalten:
Es braucht eine klare Input/Output-Syntax oder Constraints, z.B. getrennt:
  %x    = Input-Wert lesen
  %&x   = Adresse von x in ASM geben
  %=x   = Output-Temp nach ASM zurueck nach x storen

Oder eine strukturierte ASM-IR statt String-Ersetzung:
  Asm { code, inputs, outputs, clobbers }


3) Inline-ASM %&name wird nicht erkannt
Stelle:
- crates/rython_to_ir/src/codegen/stmt.rs:101-106

Was passiert:
Nach % wird nur ein Identifier-Start akzeptiert:

  if start >= bytes.len() || !is_ident_start(bytes[start] as char) {
      out.push('%');
      i += 1;
      continue;
  }

Bei "%&to_print_start" ist das Zeichen nach % ein &, also bleibt der Text
unveraendert im ASM-Code.

Warum das ein Bug ist:
Wenn %&x als "Adresse von x" gedacht ist, wird es nie ersetzt. Das sieht in
Beispielen bereits nach gewollter Feature-Syntax aus, ist aber nicht
implementiert.

Erwartetes Verhalten:
Entweder %&name offiziell parsen und die Adresse einsetzen oder die Syntax
verbieten und einen CodegenError ausgeben. Still stehenlassen ist gefaehrlich,
weil man erst sehr spaet in ASM/Backend merkt, dass es kaputt ist.


4) Operator-Overloads werden bei Calls nicht gegen Signaturen geprueft
Stellen:
- crates/rython_to_ir/src/codegen/expr.rs:333-355 fuer []
- crates/rython_to_ir/src/codegen/expr.rs:745-769 fuer unary
- crates/rython_to_ir/src/codegen/expr.rs:835-875 fuer binary
- crates/rython_to_ir/src/codegen/generator.rs:481-507 fuer Registrierung

Was passiert:
Normale Calls und Method-Calls nutzen function_signatures und pruefen:
- Funktionsname existiert
- Argumentanzahl stimmt
- Argumenttypen stimmen

Operator-Calls tun das nicht. Sie holen aus operator_functions nur:
  (mangled_name, return_ty)

Dann emittieren sie direkt:
  IrInstruction::Call {
      function_name: mangled_name,
      args: vec![lhs, rhs],
      return_type: return_ty,
  }

Warum das ein Bug ist:
Eine Operator-Methode kann eine andere Signatur haben, aber der Call wird
trotzdem erzeugt.

Beispiel:
  struct Vec2 {
      x: int,

      fn add_wrong(this: Vec2, rhs: bool) Vec2 operator + {
          return this;
      }
  }

  fn main() {
      let a: Vec2 = Vec2 { x: 1 };
      let b: Vec2 = Vec2 { x: 2 };
      let c: Vec2 = a + b;
  }

Der Operator + wird gefunden und als Vec2_add_wrong(a, b) emitted, obwohl der
zweite Parameter bool erwartet.

Erwartetes Verhalten:
Operator-Calls sollten denselben Pfad wie gen_method_call/gen_call nutzen oder
mindestens function_signatures[mangled] pruefen. Rueckgabetyp sollte ebenfalls
aus der Signatur kommen, nicht separat aus operator_functions.


5) Unaere Operator-Overloads werden falsch als binaer registriert
Stelle:
- crates/rython_to_ir/src/codegen/generator.rs:481-507

Was passiert:
preprocess_operators entscheidet unary vs binary so:

  if f.params.is_empty() {
      unary_operator_functions.insert(...)
  } else {
      operator_functions.insert(...)
  }

Warum das ein Bug ist:
Methoden haben normalerweise mindestens den this-Parameter. Ein unaerer Operator
fuer ein Struct sieht also eher so aus:

  fn neg(this: Vec2) Vec2 operator - { ... }

Das hat 1 Parameter und wird deshalb als binaerer Operator gespeichert. gen_unary_op
sucht aber in unary_operator_functions und findet ihn nicht.

Erwartetes Verhalten:
Bei Methoden muesste unary typischerweise params.len() == 1 sein (nur this),
binary params.len() == 2 (this + rhs). Alternativ muss die AST explizit
speichern, ob der Operator unary oder binary ist.


6) [] ignoriert den Index-Typ
Stelle:
- crates/rython_to_ir/src/codegen/expr.rs:333-355

Was passiert:
Der Index wird generiert, aber der Typ wird verworfen:

  let (idx_temp, _idx_ty) = self.gen_expr(idx_expr)?;

Dann wird direkt operator [](lhs, idx) aufgerufen.

Warum das ein Bug ist:
arr[true], arr["abc"] oder arr[some_struct] werden nicht am Codegen-Typecheck
abgewiesen, solange ein []-Operator fuer lhs existiert. Erst spaeter koennte
ein Backend oder Runtime-Kontrakt brechen.

Erwartetes Verhalten:
Auch [] sollte die Signatur von Struct_[](this, index) pruefen.


7) Field-Access funktioniert nicht auf temporaren Struct-Werten
Stelle:
- crates/rython_to_ir/src/codegen/expr.rs:168-220
- crates/rython_to_ir/src/codegen/expr.rs:235-269

Was passiert:
gen_field_addr ruft immer gen_left_value_addr(object) auf. gen_left_value_addr
akzeptiert nur:
- Variable
- FieldAccess
- Grouping

Andere Expressions werden InvalidExpr.

Bug-Beispiel:
  struct Point { x: int }

  fn make_point() Point {
      return Point { x: 1 };
  }

  fn main() {
      let x: int = make_point().x;
  }

make_point() hat als Ergebnis Pointer(Named("Point")). Das waere genau eine
Adresse auf einen Struct-Wert. Trotzdem scheitert gen_field_addr, weil der
object-Ausdruck kein lvalue ist.

Noch einfacheres Beispiel:
  let x: int = (Point { x: 1 }).x;

Erwartetes Verhalten:
gen_field_addr sollte zwei Faelle behandeln:
- lvalue object: Adresse aus gen_left_value_addr holen
- rvalue Pointer(Named): gen_expr(object) liefert bereits die Struct-Adresse

Wichtig:
Das ist kein "Structs brauchen Pointer"-Problem, sondern ein API-Problem in
gen_field_addr: temporare Struct-Literals/Funktionsrueckgaben sind bereits
Pointer und koennten direkt fuer GetFieldAddr genutzt werden.


8) Null ist nur Pointer(Void), nicht kompatibel mit konkreten Pointer-Typen
Stellen:
- crates/rython_to_ir/src/codegen/expr.rs, gen_null_literal
- crates/rython_to_ir/src/codegen/generator.rs:576-579 fuer const/global null

Was passiert:
null bekommt den Typ:
  Pointer(Void)

Struct-Typen werden aber als Pointer(Named("StructName")) dargestellt.

Bug-Beispiel:
  struct Node { value: int }

  fn main() {
      let n: Node = null;
  }

Wenn null als allgemeiner Null-Pointer gedacht ist, sollte das gehen. Aktuell
ist es MismatchedTypes(Pointer(Named("Node")), Pointer(Void)).

Erwartetes Verhalten:
Entweder:
- null bekommt einen speziellen Typ IrType::Null, der mit Pointer-Typen
  vereinbar ist, oder
- die Zuweisungs-/Return-/Argument-Pruefung erlaubt Pointer(Void) als Null fuer
  jeden Pointer(T).


9) Unbekannte Typnamen werden still als Named akzeptiert
Stelle:
- crates/rython_to_ir/src/codegen/generator.rs:296-319

Was passiert:
convert_to_ir_type macht:

  other => {
      let inner = IrType::Named(other.to_string());
      if self.struct_names.contains(other) {
          IrType::Pointer(Box::new(inner))
      } else {
          inner
      }
  }

Wenn "DoesNotExist" kein struct ist, wird es trotzdem Named("DoesNotExist").

Bug-Beispiel:
  fn f(x: DoesNotExist) {
  }

Der Codegen kann dafuer eine IR-Funktion mit einem unbekannten Typ bauen.

Erwartetes Verhalten:
Es braucht eine bekannte Menge definierter Typnamen, nicht nur struct_names.
Variants muessen auch registriert werden, und unbekannte Named-Typen sollten
UnknownType liefern.

Hinweis:
Dafuer muesste convert_to_ir_type wahrscheinlich Result<IrType, CodegenError>
zurueckgeben, statt immer IrType zu liefern.


10) Type::AnyTrait panict im Codegen
Stelle:
- crates/rython_to_ir/src/codegen/generator.rs:315-317

Was passiert:
  Type::AnyTrait(_) => todo!()

Warum das ein Bug ist:
Wenn der Parser/AST diesen Typ bereits produzieren kann, darf Codegen bei User-Code

11) parse_postfix panict bei ColonColon/Turbofish-Pfad
Stelle:
- crates/rython_to_ir/src/parser.rs:343-359

Was passiert:
parse_postfix hat bei TokenKind::ColonColon ein unimplemented!().

Warum das ein Bug ist:
Expr::Call besitzt bereits type_args, finish_call kann type_args annehmen, und
der Parser hat einen konkreten Token-Pfad fuer ::. Source mit dieser Syntax
panict statt einen ParseError oder sauberen Unsupported-Fehler zu liefern.

Beispiel:
  foo::<int>(1)

Erwartetes Verhalten:
Entweder korrekt parsen oder einen ParseError zurueckgeben. Kein panic.


12) type_args in Expr::Call werden im Codegen ignoriert
Stelle:
- crates/rython_to_ir/src/codegen/expr.rs:484-542

Was passiert:
gen_call bekommt type_args, benutzt sie aber nur beim rekursiven Grouping-Call.
Fuer den eigentlichen Call werden sie ignoriert.

Warum das ein Bug ist:
Wenn Call-Type-Args Teil der AST sind, dann erzeugen zwei unterschiedliche
generic calls denselben IR-Call:

  foo::<int>(1)
  foo::<bool>(true)

Beide gehen nur nach function_name "foo". Es gibt kein monomorphisiertes
Mangled-Name-Konzept und keine Pruefung, ob type_args leer sein muessen.

Erwartetes Verhalten:
Solange Generics nicht implementiert sind: type_args nicht leer -> sauberer
Unsupported/InvalidExpr-Fehler. Spaeter: type_args in Mangling/Signatur einbeziehen.


13) Name-Mangling erzeugt Kollisionen und erkennt sie nicht
Stellen:
- crates/rython_to_ir/src/codegen/generator.rs:383-403
- crates/rython_to_ir/src/codegen/generator.rs:329-381
- crates/rython_to_ir/src/codegen/generator.rs:451-470

Was passiert:
mangel verbindet Prefixe und Namen nur mit "_":

  format!("{}_{}", self.current_mangel_prefix.join("_"), name)

Kollisionen:
  fn Point_move() {}

  struct Point {
      fn move(this: Point) {}
  }

Beide heissen im IR "Point_move".

Weitere Kollision:
  struct A {
      fn B_c(this: A) {}
  }

  struct A_B {
      fn c(this: A_B) {}
  }

Beide koennen "A_B_c" ergeben.

Warum das ein Bug ist:
function_signatures ist eine HashMap. Ein Eintrag ueberschreibt den anderen.
module.functions kann trotzdem beide Funktionen enthalten. Calls koennen dann
gegen die falsche Signatur geprueft werden oder auf den falschen Namen zeigen.

Erwartetes Verhalten:
Mangled Names muessen eindeutig sein, z.B. mit Laengen-Encoding:
  S5Point4move
oder mit getrenntem Namespace im IR. Mindestens muss eine DuplicateFunction/
AmbigousFunction-Fehlermeldung entstehen.


14) Doppelte Funktionsnamen und Methodennamen werden nicht erkannt
Stellen:
- crates/rython_to_ir/src/codegen/generator.rs:329-381
- crates/rython_to_ir/src/codegen/generator.rs:451-470

Was passiert:
preprocces_functions/preprocess_method_return_types fuegen Signaturen per
HashMap::insert ein. insert ueberschreibt alte Eintraege still.

Bug-Beispiel:
  fn f(a: int) {}
  fn f(a: bool) {}

Oder:
  struct Point {
      fn move(this: Point) {}
      fn move(this: Point, dx: int) {}
  }

Ob Overloading geplant ist oder nicht: Aktuell wird es weder korrekt
unterstuetzt noch sauber verboten.

Erwartetes Verhalten:
DuplicateFunction/AmbigousFunction oder echtes Overload-Mangling nach Parametern.


15) Doppelte Struct-/Variant-Typnamen werden erst spaet/uneinheitlich erkannt
Stellen:
- crates/rython_to_ir/src/codegen/generator.rs:405-440

16) Doppelte Felder in einem Struct werden nicht bei der Definition verboten

  struct Point {

17) Lokale Variablen koennen im selben Scope doppelt deklariert werden
Stelle:
- crates/rython_to_ir/src/codegen/scope.rs:31-43

Was passiert:
insert_variable macht:

  symbols.insert(name.clone(), Variable { name, ty, addr });
  Ok(())

HashMap::insert ueberschreibt den alten Eintrag.

Bug-Beispiel:
  fn main() {
      let x: int = 1;
      let x: bool = true;
      return;
  }

Wenn Shadowing nur in inneren Scopes erlaubt sein soll, ist das falsch. Selbst
wenn Shadowing im selben Scope erlaubt sein soll, verliert man die alte Variable
ohne klare Semantik.

Erwartetes Verhalten:
Vor insert pruefen, ob symbols im aktuellen Scope den Namen schon enthaelt.


18) Globals/Consts werden nur gegeneinander geprueft, nicht gegen Functions/Types
Stelle:
- crates/rython_to_ir/src/codegen/generator.rs:509-555

Was passiert:
process_consts_and_globals prueft nur:
  module.constants
  module.globals

Nicht geprueft werden:
- globale Funktionen
- Struct-/Variant-Namen
- mangled Methodennamen

Bug-Beispiel:
  const f: int = 1;
  fn f() {}

Oder:
  struct Point { x: int }
  global Point: int = 0;

Wenn Namespaces nicht absichtlich getrennt sind, ist das eine Kollision.
Wenn Namespaces getrennt sein sollen, muss das im Resolver/IR eindeutig
modelliert werden.


19) Operator-Overloads ueberschreiben sich bei gleichem Struct+Operator
Stelle:
- crates/rython_to_ir/src/codegen/generator.rs:481-507

Was passiert:
operator_functions und unary_operator_functions sind HashMaps keyed by:
  (struct_name, operator_string)

Wenn ein Struct zwei + Operatoren hat, gewinnt der letzte insert.

Bug-Beispiel:
  struct Vec2 {
      fn add_int(this: Vec2, rhs: int) Vec2 operator + { ... }
      fn add_vec(this: Vec2, rhs: Vec2) Vec2 operator + { ... }
  }

Aktuell kann das nicht als Overload funktionieren, wird aber auch nicht verboten.

Erwartetes Verhalten:
Entweder Parametertypen in den Key aufnehmen oder Duplicate/Ambiguous Operator
melden.


20) Rechter Operator-Fallback vertauscht Operanden semantisch riskant
Stelle:
- crates/rython_to_ir/src/codegen/expr.rs:862-875

Was passiert:
Wenn links keinen Operator hat, aber rechts ein Struct mit Operator, wird
folgendes gemacht:

  args: vec![temp_id_2, temp_id_1]

Also a + b wird zu B_plus(b, a).

Warum das ein Bug sein kann:
Bei nicht-kommutativen Operatoren ist das nicht automatisch korrekt. Es ist nur
dann richtig, wenn die Sprache explizit "reverse operator dispatch" so definiert.

Beispiel:
  10 - my_number

Wenn my_number einen operator - hat, bekommt die Methode (my_number, 10). Das
ist nicht dasselbe wie 10 - my_number, ausser die Methode ist bewusst als reverse
operator implementiert.

Erwartetes Verhalten:
Entweder kein rechter Fallback oder separate reverse operator Semantik.
Mindestens Signaturcheck und klare Spezifikation.


21) && und || sind eager statt short-circuit
Stelle:
- crates/rython_to_ir/src/codegen/expr.rs:835-895

Was passiert:
gen_binary_op erzeugt immer zuerst lhs und rhs:

  let (temp_id_1, ir_type_1) = self.gen_expr(lhs)?;
  let (temp_id_2, ir_type_2) = self.gen_expr(rhs)?;

Danach wird Binary And/Or emitted.

Warum das ein Bug ist, falls die Sprache normale &&/|| Semantik will:
Bei short-circuit darf rhs nicht ausgewertet werden, wenn das Ergebnis schon
durch lhs feststeht.

Bug-Beispiel:
  fn boom() bool {
      asm { invalid_or_side_effect }
      return true;
  }

  fn main() {
      let x: bool = false && boom();
  }

boom() wuerde aktuell trotzdem generiert/ausgefuehrt.

Erwartetes Verhalten:
&& und || brauchen Control-Flow-Bloecke und wahrscheinlich Phi/temporary store
oder ein anderes IR-Konzept fuer den Ergebniswert.


22) Infinite loop erzeugt immer einen loop_end Block
Stelle:
- crates/rython_to_ir/src/codegen/stmt.rs:272-295

Was passiert:
gen_loop erstellt immer:
  loop_body
  loop_end

Auch wenn es im Loop keinen break gibt. Bei non-void Funktionen kann loop_end
dann leer und ohne Terminator bleiben, was finish_blocks als MissingTerminator
meldet.

Bug-Beispiel:
  fn main() int {
      loop {
      }
  }

Falls endlose Loops als nie-returning gelten sollen, sollte das akzeptiert
werden. Aktuell entsteht ein unerreichbarer, aber terminatorloser Block.

Erwartetes Verhalten:
loop_end nur erzeugen, wenn ein break erreichbar ist, oder unreachable/noreturn
Terminators einfuehren.


23) Unreachable Code nach return/break/continue wird als harter Fehler behandelt
Stellen:
- crates/rython_to_ir/src/codegen/stmt.rs:162-170
- crates/rython_to_ir/src/codegen/generator.rs:71-95

Was passiert:
gen_block generiert alle Statements weiter. Wenn der aktuelle Block schon einen
Terminator hat, schlagen weitere Instructions mit CodeAfterTerminator fehl.

Beispiel:
  fn main() {
      return;
      let x: int = 1;
  }

Das ist aktuell CodegenError::CodeAfterTerminator.

Warum das ein Bug sein kann:
Viele Sprachen erlauben unreachable code mit Warning oder eliminieren ihn. Wenn
Rython unreachable code absichtlich verbieten will, ist es kein Bug. Dann sollte
es aber bewusst als Sprachregel dokumentiert sein.

Fix-Richtung falls erlaubt:
gen_block nach Terminator abbrechen oder unreachable Statements ignorieren/diagnostizieren.


24) Void-Calls produzieren trotzdem einen Wert-Temp
Stellen:
- crates/rython_to_ir/src/codegen/expr.rs:530-541
- crates/rython_to_ir/src/codegen/stmt.rs:39-79
- crates/rython_to_ir/src/ir.rs:110-116

Was passiert:
IrInstruction::Call hat immer temp_id: TempId, auch bei return_type Void.
gen_return hat Sonderlogik, damit return foo(); bei void nicht Ret(Some(temp))
wird.

Warum das ein Bug/Designleck ist:
Void wird damit als Expression-Wert weitergereicht. In vielen Kontexten wird es
ueber Typchecks abgefangen, aber das IR modelliert trotzdem einen Rueckgabewert,
den es semantisch nicht gibt.

Beispiel:
  fn foo() {}

  fn main() {
      foo();        // ok
      return foo(); // wird speziell repariert
  }

Erwartetes Verhalten:
Call sollte fuer void entweder keinen temp haben:
  temp_id: Option<TempId>
oder void calls sollten nur als Statement erlaubt sein.


25) Struct-Literal allociert Speicher, aber Ownership/Lifetime ist unklar
Stelle:
- crates/rython_to_ir/src/codegen/expr.rs:360-417

Was passiert:
gen_struct_literal macht Alloca fuer den Struct und gibt Pointer(Named(...))
zurueck:

  Alloca { temp_id: struct_addr, ty: Named("Point") }
  Store field values
  Ok((struct_addr, Pointer(Named("Point"))))

Warum das relevant ist:
Das ist fuer p.x und Methoden gut, weil man eine Adresse braucht. Aber bei
Rueckgabe aus Funktionen ist unklar, ob dieser Alloca-Speicher ueber den
Funktionsrahmen hinaus gueltig ist.

Bug-Beispiel:
  fn make_point() Point {
      return Point { x: 1 };
  }

Wenn Alloca Stack-Speicher im Funktionsframe bedeutet, gibt make_point einen
Pointer auf lokalen Speicher zurueck. Das waere ein dangling pointer.

Erwartetes Verhalten:
Entweder:
- Structs by value im IR sauber kopieren,
- return-slot/sret verwenden,
- heap allocation fuer escaped Struct-Literals,
- oder klare Lifetime-Regel im Backend.

Hinweis:
Das ist kein reiner "noch nicht implementiert"-Punkt, weil Struct-Literals und
Struct-Returns bereits als Feature verbunden sind.


26) Parameter-Handling kopiert Struct-Pointer nur in einen lokalen Slot
Stelle:
- crates/rython_to_ir/src/codegen/generator.rs:150-214

Was passiert:
Jeder Parameter wird:
1. allociert,
2. per LoadParam geladen,
3. in den lokalen Slot gespeichert.

Bei Struct-Typen ist parameter_type bereits Pointer(Named(...)). Das heisst:
Es wird der Pointer-Wert gespeichert, nicht der Struct-Inhalt kopiert.

Warum das ein Bug sein kann:
Wenn die Sprache Struct-Parameter by value meint, sind Methoden/Funktionen
aktuell by reference/by pointer. Mutationen an this.field koennen das Original
veraendern.

Beispiel:
  struct Point { x: int }

  fn change(p: Point) {
      p.x = 10;
  }

  fn main() {
      let p: Point = Point { x: 1 };
      change(p);
      // p.x ist wahrscheinlich 10, falls Pointer-Semantik durchgereicht wird.
  }

Erwartetes Verhalten:
Entweder Structs offiziell reference types machen oder bei by-value eine Kopie
des Struct-Speichers erzeugen.

31) Interne expect()/panic-Pfade im Codegen
Stellen:
- crates/rython_to_ir/src/codegen/generator.rs:62-69 jump_to_block
- crates/rython_to_ir/src/codegen/scope.rs:27-43 enter/exit/insert

Was passiert:
Mehrere interne Invarianten benutzen expect:
  expect("block does not exist")
  expect("No active scope")
  expect("No active scope to exit")

Warum das aufgenommen ist:
Wenn ein User-Programm einen Generatorzustand triggert, der diese Invariante
bricht, crasht der Compiler statt CodegenError zu liefern. Aktuell sind diese
Pfade eher interne Bugs als direkt reproduzierbare User-Bugs, deshalb niedrigere
Prioritaet als die Punkte oben.


Priorisierte Fix-Reihenfolge:
1. Variant-Lowering auf InitVariant umstellen.
2. Operator-Calls ueber function_signatures validieren; unary Klassifizierung fixen.
3. Name-Duplikate/Kollisionen im Preprocessing ablehnen.
4. Field-Access fuer rvalue Struct-Pointer erlauben.
5. Null-Typ als speziellen kompatiblen Typ modellieren.
6. Inline-ASM Input/Output/Address-Syntax sauber definieren.
7. Duplicate checks fuer locals, fields, typedefs, functions.
8. Panic-Pfade durch Errors ersetzen.
9. Warnings/Typos aufraeumen.
